# Backend Implementer Agent Specification

## Domain
Services, workers, procedures, validation schemas, error handling, database operations.

## Purpose
Implement backend work following plans generated by planning agents. Reads the plan context package and implements compliant code.

## How to Use This Agent

When given an implementation plan:

1. **Read the plan** - Contains all context, file paths, patterns
2. **Read PR review agents** - Compliance standards linked in plan
3. **Read pattern references** - Example files linked in plan
4. **Implement following plan** - Follow code templates in plan
5. **Verify acceptance criteria** - Check each requirement in plan
6. **Update beads** - Add milestone comments

## Compliance Standards (From PR Review Agents)

- **Service Agent**: `.claude/skills/pr-review-agent-team/agents/services.md`
  - Extend `BaseService`
  - Use `this.obs` for logging
  - Use `this.handleError()` for error wrapping
  - Throw specific errors (NotFoundError, ConflictError, BusinessLogicError)
  - No business logic in workers

- **Worker Agent**: `.claude/skills/pr-review-agent-team/agents/workers.md`
  - Use `procedure()` wrapper for all routes
  - No direct database access in routes
  - Delegate to services via `ctx.services`
  - Correct HTTP status codes (201 for POST, 204 for DELETE)
  - Input validation via Zod schemas

- **Database Agent**: `.claude/skills/pr-review-agent-team/agents/database.md`
  - Use `scopedNotDeleted(table, creatorId)` for all queries
  - Use `dbWs.transaction()` for multi-step operations
  - Soft delete pattern (deletedAt column)
  - Ownership verification before mutations

- **Security Agent**: `.claude/skills/pr-review-agent-team/agents/security.md`
  - Auth policy on state-changing endpoints
  - Rate limiting configuration
  - Input validation with Zod
  - No PII in logs

## Key Patterns to Follow

### Service Method
- Extend BaseService pattern
- Use scoped queries with creatorId/organizationId
- Throw specific errors (not generic Error)
- Log via this.obs
- Wrap errors with this.handleError

### Worker Route
- Use procedure() wrapper
- No business logic in routes
- Delegate to ctx.services
- Set correct status codes (201 for POST, 204 for DELETE)
- Include auth policy and rate limiting

### Database Operations
- Use scopedNotDeleted for queries
- Use dbWs for transactions
- Soft delete via deletedAt
- Verify ownership before mutations

### Validation Schema
- Define in packages/validation/src/schemas/
- Reuse primitives from primitives.ts
- Include business logic refinements

## Anti-Patterns to Avoid

- ❌ Don't put business logic in workers
- ❌ Don't query by ID alone (data leak)
- ❌ Don't use dbHttp for transactions
- ❌ Don't use physical DELETE
- ❌ Don't throw generic Error
