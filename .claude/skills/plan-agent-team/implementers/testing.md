# Testing Implementer Agent Specification

## Domain
Unit tests, integration tests, factories, coverage, test isolation.

## Purpose
Implement test coverage following plans generated by planning agents. Ensures compliance with Testing PR review agent standards.

## How to Use This Agent

When given an implementation plan:

1. **Read the plan** - Contains test requirements and coverage targets
2. **Read PR review agent** - Testing Agent compliance standards
3. **Read pattern references** - Example test files
4. **Implement following plan** - Use factories, test both paths
5. **Verify acceptance criteria** - Coverage, isolation, proper cleanup

## Compliance Standards (From PR Review Agents)

- **Testing Agent**: `.claude/skills/pr-review-agent-team/agents/testing.md`

**CRITICAL Rules**:
- [CRITICAL] Critical paths covered (create, read, update, delete, errors)
- [WARN] Tests use factories from `@codex/test-utils`
- [WARN] Both success and error paths tested
- [WARN] Clean up test data in `afterEach` or use transactions
- [INFO] Test isolation (each test independent)

## Key Patterns to Follow

### Test Organization
- Co-locate tests in `__tests__/` subdir
- Use `describe()` blocks for methods
- Use `beforeAll`/`afterEach` for setup/teardown
- Name files `*.test.ts` matching source

### Factory Usage
- Use `createTestContent()`, `createTestUser()` etc.
- Custom data via overrides parameter
- Factories create valid data by default

### Database Testing
- Use `setupTestDatabase()` for DB setup
- Use `dbWs` for test transactions
- Clean up test data in `afterEach`

### Test Coverage
- Success path: normal operation
- Error paths: 404, 403, validation errors
- Edge cases: boundary conditions

## Anti-Patterns to Avoid

- ❌ Don't use hardcoded test data (use factories)
- ❌ Don't test only success paths
- ❌ Don't skip cleanup in tests
- ❌ Don't use shared state between tests
