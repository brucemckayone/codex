## Key insight — first commerce audit surfaces a new failure class

Commerce is Payment-API-adjacent. It changes the threat model. Media-review iterations asked *"is this accessible under reduced-motion?"*; commerce-review iterations add *"does a rejected payment try to render an error message that includes the card fingerprint?"* The new ref 05 §Checkout & payment flows subsection codifies five patterns observed in this batch:

1. **State-machine live-regions** — `pending → complete → fallback` transitions each need announcement. The CheckoutSuccess card now has both an outer `aria-live="polite"` wrapper (for state swaps) and an inner pending spinner with `role="status" aria-live="polite"` (for initial load). Nested live regions are acceptable per WAI-ARIA APG — the inner region wins on announce when both are present.
2. **Generic error messaging** — NEVER echo `err.message` from Stripe or any payment provider directly. The message may contain card-prefix substrings, customer account IDs, or internal reference tokens. Use Paraglide keys (`m.subscription_checkout_error()`). Log the original via observability/console for debug, render the generic for the user.
3. **Retry-exhaustion announcement** — after N failed polling cycles, the fallback UI swap needs to announce ("We couldn't confirm your payment in time — please check email"). Simply swapping the DOM is silent to screen-readers without the `aria-live` wrapper.
4. **Buy/Subscribe button triple-state** — `aria-busy={loading} disabled={loading}` both. `aria-busy` is the semantic announcement; `disabled` prevents double-submission. Missing either creates a UX hole.
5. **Title-attribute anti-pattern** — `title="{fullName}"` as the sole disclosure for a truncated tier name violates WCAG 1.4.13 (content on hover / focus must also be keyboard-accessible). Either remove the truncation (badge sizes to content — simplest) or wrap in a Melt UI `createTooltip` primitive (activates on both focus and hover).

This is ref 05's 8th subsection in the §7 Forms region. The write-then-consume pattern applied again: the skill patch landed during Phase-A triage, the implementation agent referenced it in Phase B to pattern the fix. Proven approach across iter-08, iter-11, iter-12, iter-14.

This would also catch architectural debt (dead components) as a side-effect of the audit process.

## Key insight — duration-token ceiling found

The CheckoutSuccess spinner had `animation: spin 1s linear infinite`. Mapping to a motion token revealed the ceiling: the largest duration token in `apps/web/src/lib/styles/tokens/motion.css` is `--duration-slower` (500ms). Nothing for 1000ms.

The implementation agent mapped `1s → --duration-slower` (500ms) as closest-available. The resulting spinner spins twice as fast as intended — acceptable for a loading indicator, not necessarily right for other long-form animations (page transitions, fade-ins on scroll).

**Iter-15 candidate**: `--duration-longest` token at 800ms–1000ms. Motion-ladder semantic stop between "slower" (500ms, punctuated UI feedback) and "infinite" (keyframe loops). Not urgent — checkout works correctly — but worth a P3 bead so the gap doesn't get patched with `1s` literals next time someone needs a long animation.
