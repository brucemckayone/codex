## Key process insight — bidirectional drift

Iter-04 and iter-05 together prove the loop catches **both drift directions**:

| Direction | Example | Iter | Fix |
|---|---|---|---|
| Skill ahead of code | Ref 10 §5 C4 cited `+layout.svelte` but code moved to `BrandEditorMount.svelte` | iter-04 | Patched skill to match code |
| Code ahead of skill | Ref 08 invented `axeCheck` helper that never existed | iter-05 | **Patched skill to match code** — but this time the "lie" was always there |

The second case is subtler: the skill wasn't stale, it was **fabricated** at authoring time. No agent had ever run `axeCheck` because it never existed — the import would have errored immediately. This is a fabrication-catching pattern: if the skill claims an API exists, and no test in the repo uses it, the claim is suspect.

**Durable skill process improvement (candidate for SKILL.md §7)**: before an iteration closes, grep the skill's cited-canonical APIs against the codebase. Any claim that has zero consumers is a lie. Worth adding to the per-iteration workflow.
