## The P0

The first P0 of the loop. Verified by direct read across four files:

1. `packages/worker-utils/src/procedure/multipart-procedure.ts:282-309` — MIME string + size check only. No magic numbers, no content inspection.
2. `workers/organization-api/src/routes/settings.ts:282-314` — handler passes raw `{buffer, mimeType, size}` to `uploadLogo`.
3. `packages/platform-settings/src/services/branding-settings-service.ts:254-310` — JSDoc claims "validation handled by `validateLogoUpload()`" at lines 248-251; **no code calls `validateLogoUpload()`** (grep across entire repo confirms zero callers outside tests); buffer is written to R2 at line 279 with 1-year `Cache-Control`.
4. `packages/image-processing/CLAUDE.md:64` explicitly: **"MUST sanitize ALL SVG uploads with `sanitizeSvgContent()` — unsanitized SVGs are XSS vectors"**.

The CLAUDE.md rule is violated, the JSDoc is aspirational, and the code ships an XSS vector.

**Exploit**: an org admin (or compromised admin account) uploads an SVG with `<script>` inside. The SVG lives at `cdn.codex/logos/{orgId}/logo.svg` with 1-year cache. When the SVG is loaded directly (not via `<img>`) — in embed widgets, admin previews, phishing links, `<object>` consumers — the script executes with same-origin privileges on the CDN domain.

**Fix**: call `sanitizeSvgContent()` in `uploadLogo` for `image/svg+xml` MIME types before the `r2.put` call. Minimal diff. Pattern matches how `ImageProcessingService.service.ts:365` already does it for the image-processing path.

## Skill patches applied

**None this iteration.** Consistent with iter-018–iter-023. But this iteration generates **two strong skill-patch candidates**:

1. **Binary upload security checklist** for reference 10 (brand-editor). Currently no mention of multipart upload gotchas. Would have caught `06ygy` at authoring time. Queue post-fix.

2. **"JSDoc != contract" anti-pattern** for reference 03 (components) or a new meta-rule. When a service's JSDoc says "caller must do X", that's an implicit contract the type system doesn't enforce. `uploadLogo`'s JSDoc claimed sanitization was done upstream; reality was it wasn't. The safer pattern: move the validation INTO the service, or make the input type carry proof of validation (branded types).
