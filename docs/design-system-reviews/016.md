## Key insight ‚Äî 4th P0 teaches the chart-a11y contract

Loop blockers so far: iter-08 OklchColorArea (2D keyboard contract), iter-13 VideoPlayer captions regression, iter-15 MediaUpload `role="button"` div, iter-16 RevenueChart hover-only tooltip. All four are custom-interactive / visual-info surfaces where WCAG 2.1.1 (Keyboard) or 1.3.1 (Info and Relationships) was being bypassed by a sighted-mouse assumption.

The correct chart-a11y pattern is two orthogonal contracts:
1. **The graphic** ‚Üí `<svg role="img" aria-label="‚Ä¶summary with total/peak/low‚Ä¶">`
2. **The data** ‚Üí sibling `<table class="sr-only">` mirroring the data array

This is cheaper than focusable-bar alternatives and also works for print + voice output. The new ref 05 ¬ßData visualisation subsection codifies both patterns with canonical code + "chart colours must use semantic tokens, never raw palette" (iter-15 R15-candidate reinforcement).

## Key insight ‚Äî R13 100% miss rate in a fresh batch

All 3 files in the iter-16 batch lacked a `class?: string` prop. R13 has been a hard rule since iter-13 (promoted on 3-iteration recurrence per ¬ß7). The rule text is clear. Yet new components keep shipping without it.

Possible explanations:
- Component authors copy from `SkeletonContentCard` era before R13 was promoted.
- Svelte 5 `$props()` destructuring with `class:` alias is slightly awkward syntax ‚Äî friction at authorship time.
- Tests and typecheck don't enforce it ‚Äî R13 is a soft linting concept, not a type error.

**Not yet R-promotion-candidate** (already IS R13) ‚Äî but worth considering a `scripts/check-class-prop.ts` linter that greps for components with `<root-element class="‚Ä¶">` but no `class` in the `interface Props`. Costs ~1hr to write, catches every future instance. File as a tools bead for iter-18 after iter-17 content-form batch.
