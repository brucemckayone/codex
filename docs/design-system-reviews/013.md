## Key insight — the P0 blocker teaches a process rule

Iter-011 added a `captions` prop on VideoPlayer (Codex-unkj8 fix). The fix compiled, passed MCP verification, and landed in two sequential iterations without detection. The reason: **iter-011's MCP gate verified the PROP accepted captions, not that the CONSUMER passed them.** End-to-end verification requires the primary consumer to be in the audit batch, not just the producer.

iter-013's ContentDetailView inclusion was deliberate ("bridges from media to content surfaces"). That bridge-file choice caught the regression inside 2 iterations rather than letting it rot. Without it, the blocker could have persisted until a user reported missing captions in production.

**Process rule candidate for future iterations**: When a prop shape changes on a component, file a dependency-linked follow-up bead against the primary consumer, OR include the consumer in the same batch. Otherwise the fix is technically landed but semantically unshipped.

## Key insight — sibling-composable pattern established

Iter-012 extracted `createMediaKeyboardHandler` (real `HTMLMediaElement`). Iter-013's Waveform audit revealed the helper doesn't fit canvas sliders (no media element, `onSeek(time)` callback). The wrong move would have been to widen `createMediaKeyboardHandler`'s type signature to cover both cases. The right move — taken by the implementation agent — was to add a **sibling** `createSliderKeyboardHandler` in the same file.

Both composables cohabit `$lib/components/media-a11y/keyboard.svelte.ts`. Future maintainers see both at once and understand the branch point: media elements mutate `currentTime` directly; canvas sliders emit callbacks. This is the canonical shape for extending primitive libraries — siblings over generics when the underlying state models differ.

## Key insight — live composable verification

The MCP gate evolved one more notch. Previously: "a11y tree renders correctly" (iter-8 onward), "keyboard scoping verified bidirectionally" (iter-11). Iter-013 adds: **live browser synthetic-KeyboardEvent test on the composable itself**, firing each of 7 APG-contract keys and asserting each produces exactly one correct `onSeek` call.

That's the strongest verification format we've had. For extracted primitives (not just components), the test surface is the API, not the rendered DOM. The implementation agent wrote a temporary test harness inline during verification — a pattern worth naming: **API harness verification** for pure-logic composables.
