## Key insight — third P0 teaches the file-input contract

The loop's first two P0 blockers surfaced WCAG 2.1.1 failures on custom interactive widgets: iter-08 OklchColorArea (2D keyboard contract absent) and iter-13 VideoPlayer captions regression (prop added but not wired by consumer). Iter-15's P0 is the same class — a `<div role="button" tabindex="0">` wrapping a hidden file input on MediaUpload. The custom keybinding handled Space/Enter but not drag+activate semantics, and the `aria-hidden="true" tabindex="-1"` on the actual `<input type="file">` made it SR-invisible.

The correct fix is the `<label>`-over-`<input>` pattern (native file-input's idiomatic shape since 1997). The new skill subsection **ref 05 §File input + drag-drop upload** codifies 5 patterns derived from this fix:

1. **Label-over-input** — never `<div role="button">` over a hidden file picker. Use `<label for="…">` wrapping the visible click surface; the `<input type="file">` lives inside with `class="sr-only"` (no `aria-hidden`).
2. **Queue aria-live** — drag accept/reject, queue mutations, progress, errors all announce via one outer `role="status" aria-live="polite"` wrapper; per-item errors escalate to `role="alert"`.
3. **Rejection tracking** — invalid files should NOT be silently dropped; push to a `rejections` $state array and render them in the same live region (announces automatically).
4. **Generic error messaging** — raw XHR / remote-function errors may leak HTTP status bodies or R2 presigned-URL fragments; map to a generic Paraglide message like `m.media_upload_error()`, log the real error via observability.
5. **`aria-label` uniqueness per row** — a Remove button in each queue row needs `aria-label={`Remove ${item.file.name}`}` — bare `aria-label="Remove"` creates N identical AT targets.

This subsection is ref 05's 9th §7 Forms subsection (§Checkout & payment flows from iter-14 was the 8th). Write-then-consume applied for the 9th time: patch landed during triage, implementation agent cited it directly while writing the fix.

## Key insight — svelte-autofixer surfaced 2 novel a11y rules

The implementation agent's MCP-verification step caught two things the skill didn't document:

1. **`aria-expanded` on `role="radio"` is invalid** — ARIA 1.2 doesn't allow it on the radio role. When composing radio-selection with disclosure (e.g. "Paid" radio reveals price panel), put `aria-expanded` / `aria-controls` on the wrapping `<label>` or on a fieldset container, not the `<input>` itself.
2. **`<section>` has implicit `role="region"`** — adding explicit `role="region"` is redundant and autofixer flags it.

(1) is novel enough to warrant a skill patch — added as a follow-up to ref 05 §4 Custom radiogroup with a canonical wrong/right code example. (2) is small polish caught cleanly by autofixer without needing a skill entry.

**Process lesson**: svelte-autofixer is doing durable a11y work. The implementation phase's autofixer invocation is not just "does the file parse" — it's actively surfacing ARIA-authoring gaps the review agent missed in static inspection. Worth including autofixer findings in the audit doc going forward.
