## Key insight — write-then-consume, iter-008 pattern replicated

Iter-008 was the first time a skill patch landed mid-triage and was immediately consumed by the implementation agent. Iter-011 replicates: the new ref 05 §"Media Elements" subsection was patched during triage, and the implementation agent's fix sequence followed §1→§6 verbatim:
- §1 → `aria-label` on `<video>` ✓
- §2 → full `<track>` attribute set + CC toggle button ✓
- §3 → keyboard shortcuts scoped to `.video-player-wrapper` ✓
- §4 → `role="status"` wrapping with nested `role="alert"` ✓
- §5 → 9-rule `@media (prefers-reduced-motion: reduce)` block ✓
- §6 → `role="presentation"` removed from tap-zone ✓

This cadence works when patches are *specific* (exact ARIA trees, exact keyboard-scoping patterns) rather than *exhortative*. For hotspots, it compresses review → patch → implement from 2 iterations into 1.

## Key insight — a11y tree snapshot is the strongest MCP verification

The MCP gate has been exercised in 6 consecutive iterations now, but iter-011 produced the **strongest single piece of evidence** — an accessibility tree snapshot showing:

```
region "Video player" [role=region, aria-label, tabindex=-1]
  status atomic live="polite"
    alert atomic live="assertive"
      StaticText "Failed to load video…"
    button "Try Again"
```

This verifies not just that the rule compiles, but that the **ARIA relationships render correctly**: outer status region is polite (progress updates) while the inner alert span escalates to assertive without interrupting the polite context. That's the tricky pattern — DevTools a11y tree is the only way to confirm it practically.

Adding this insight to the MCP gate expectations: for a11y-heavy PRs, request an a11y tree snapshot, not just "focus ring visible" confirmation.

## Key insight — two kinds of `svelte-ignore`

Iter-011 saw both kinds in the same component:
- **Silent suppression (bad)**: the original time-pill `svelte-ignore a11y_click_events_have_key_events` was hiding a real bug (`<div role="button" tabindex="0">` instead of native `<button>`). Fix was to delete the ignore AND the underlying `div` wrapper.
- **Documented exception (good)**: the post-fix wrapper `<div role="region">` needed an `a11y_no_noninteractive_element_interactions` ignore because the skill explicitly prescribes keyboard shortcuts on the wrapper (not on focusable descendants). The ignore is accompanied by a skill-citation comment.

The distinction matters for future reviews: `svelte-ignore` without a reasoning comment is a smell; `svelte-ignore` with a skill/reference citation is a design decision.

## Keyboard-scoping verification

The load-bearing claim of iter-011 ("global hijack → wrapper-scoped shortcuts") was verified via evaluate_script:
- Dispatch `keydown { key: ' ' }` on a `<button>` **outside** `.video-player-wrapper` → wrapper handler did NOT fire. ✓
- Dispatch on a `<button>` **inside** `.video-player-wrapper` → wrapper handler DID fire. ✓

Scoping works in both directions. This is the test pattern for any "does my event handler respect element scope?" fix.
