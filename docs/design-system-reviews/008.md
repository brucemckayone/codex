## Key insight — write-then-consume in a single iteration

Iters 01–07 followed the pattern: review finds pattern-level learning → patch the skill → *next* iteration's agent benefits. Iter-008 changed the cadence: patch lands during triage, then the implementation agent reads the updated skill *before* writing the fix.

This worked because:
- The patches were *specific* (exact keyboard contract, exact RAF pattern) rather than *exhortative* ("be accessible")
- The implementation agent was instructed to re-read the skill after invoking `/design-system`, not just to use memory from prior sessions
- The fixes cited the new sections explicitly (e.g. "per new ref 05 §Custom 2D slider" in the a11y-tree verification)

Net: the review->patch->implement loop compresses from 2 iterations to 1. For hotspots, this means better fixes; for cold spots, the prior "fix next time" cadence is fine.
