## Key insight — iter-16 regression caught in iter-17

`apps/web/src/lib/components/ui/Icon/CopyIcon.svelte` shipped to main in iter-16's commit `a4e7422f` with two trailing XML tags (`</content>` and `</invoke>`) that leaked into the file from the implementation agent's Write tool output. The SvelteKit compiler treats this as a fatal parse error — the **dev server returned 500 on every route** for anyone who pulled main after `a4e7422f`.

Iter-17's implementation agent discovered this when trying to MCP-verify studio routes; `pnpm dev` was up but every page returned 500. The subagent honestly reported the gap instead of silently skipping verification. Fix: overwrote the file with the correct content (10 lines; tags 11-12 removed).

**Process lesson — MCP dev-server gap caused the regression.** Iter-16 skipped live verification because no dev server was running. Static + typecheck + autofixer all passed because they don't compile the full Svelte file — they parse the script block. The XML tags appeared only below the closing `</IconBase>` in the template section, which Svelte's parser flagged at compile time.

**Hardening candidate for iter-18+ MCP gate**: the iteration-close checklist should include `pnpm --filter web build` (or equivalent compile) when any `.svelte` file is created. Unlike typecheck, the build compiles templates and catches parse errors that typecheck misses. Adding ~15s per iteration, catches a real class of regressions.

## Key insight — `--space-32` undefined-token discovery

The implementation agent noticed `MediaSection.svelte:280` referenced `var(--space-32)` — and spacing.css only defines `--space-0` through `--space-24`. The value resolves to `initial` (CSS custom property default), silently breaking `min-height` on the affected elements.

Grep found 4 callers app-wide:
- `MediaSection.svelte:280` — fixed iter-17 (swapped to `--space-24` with explainer comment).
- `pricing/+page.svelte:1952` — still broken.
- Studio content-edit page — still broken.
- Studio content-new page — still broken.

**This is R1 hiding in plain sight.** R1 says "never hardcode values where tokens exist" — but consumers can also reference tokens that DON'T exist, with no compile-time error. A token-layer lookup validator would catch this: grep every `var(--[a-z0-9-]+)` reference, cross-check against the union of declared custom properties in `styles/tokens/`. Zero-cost CI check.

Filed `Codex-jwq8a` as carry-forward. Either extend the scale (add `--space-32`) or sweep the 3 remaining callers. Option (a) extends the scale consistently and is the lower-friction path.

## Key insight — RichTextEditor inverse-R13 is a roadmap marker

When wiring ContentDetails title field for validation SR (aria-invalid + aria-describedby), the description field uses `RichTextEditor` — which doesn't accept ARIA props for forwarding to its inner contentEditable element. The implementation agent didn't silently drop the wiring; they documented the seam with a comment and kept the `<p id="description-error">` in place for the day RichTextEditor extends.

**Inverse-R13 pattern named** (iter-08 canonical): when a primitive doesn't expose a prop, either add the prop (preferred) OR leave a comment explaining why callers can't style/augment it. The implementation agent took option (b) — correct for scope.

RichTextEditor solo review is the obvious next step for the ARIA forwarding. Filed `Codex-xopdh` as a roadmap marker, scheduled for iter-19 or iter-20. Iter-18 will audit ContentForm.svelte solo.
