## Synthesis

### Clusters
- **Stripe / Connect monetisation gating** (`connect-live-vs-test`, `tier-gate-connect`) — Everything touching Stripe mode and Connect-readiness gating on the write path. connect-live-vs-test is a prod secret misconfiguration (sk_live_ key deployed to ecom-api-production while the launch bar calls for TEST mode; per-mode webhook whsec_ secrets are implicated in the same rotation). tier-gate-connect is the backend correctly hard-gating tier creation on Connect readiness (requireActiveConnect) but the UI failing to gate, so the block surfaces as an opaque 'failed to save tier' rather than proactive guidance.
- **Subscription surfacing & monetisation empty-states** (`subscribe-component-shows-without-subscription`, `subscribers-empty-state`, `sales-payouts-empty-state-and-layout`) — Read-path / discovery surfaces that fail to reflect the org's actual monetisation state. subscribe-component-shows-without-subscription renders the Subscribe CTA/sticky bar with no check for active tiers, enableSubscriptions, or Connect. subscribers-empty-state and sales/payouts empty-states lack condition-aware, prerequisite-aware first-creator guidance (Connect not set up → no tiers → no subscribers/sales/payouts). All three share the pattern of not consulting connectStatus/tier-count before rendering monetisation UI.
- **Studio layout & UI primitives** (`studio-sidebar-scrollbar`, `sales-payouts-empty-state-and-layout`) — Pure presentation defects in the studio shell. studio-sidebar-scrollbar exposes an un-styled native scrollbar on the hover-expanded rail (overflow-y:auto with no scrollbar-width/webkit styling; token-driven fix). The sales/payouts pages are capped by hardcoded max-width (1200px/1280px, also a design-token violation) so they don't fill the studio column like their siblings. (sales-payouts-empty-state-and-layout spans two clusters: its width defect is UI/layout, its empty-state defect is monetisation surfacing.)
- **Auth / password-reset email** (`forgot-password-email`) — The web app POSTs forgot-password to /api/auth/forget-password, an endpoint that does not exist in the installed BetterAuth 1.4.11 (real endpoint is /api/auth/request-password-reset). The 404 is silently swallowed and a fake success is returned, so no reset email ever sends. Long-standing latent bug, not a 4-day regression. Same stale literal is baked into the rate-limiter allowlist.
- **Content / media rendering** (`content-500-liberty`) — The content-detail SSR load imports generateHTML from @tiptap/html (browser build) instead of @tiptap/html/server. The browser build throws when window is undefined under workerd, so any public/free content with a TipTap JSON body 500s. On this org only the audio item 'liberty' matches (free + contentBodyJson present) and reaches the eager public render, which is unguarded (.catch missing) so it takes down the whole page.

### Cross-cutting root causes (one fix → multiple bugs)
- **Monetisation UI surfaces do not consult Stripe Connect readiness / active-tier state before rendering. The backend already treats Connect + active tiers as the gate for taking money (TierService.requireActiveConnect; tiers can only exist post-Connect), but the frontend renders monetisation CTAs and write actions unconditionally. One coherent fix — thread the already-available connectStatus (getConnectStatus) and tier-count into a shared 'canAcceptSubscriptions'/'connectReady' derivation and gate every monetisation surface on it — resolves the tier write-path block and the public/empty-state read-path surfacing together.** → `tier-gate-connect`, `subscribe-component-shows-without-subscription`, `subscribers-empty-state`
  - Shared fix: Introduce a single Connect-readiness + active-tiers signal (connectStatus.chargesEnabled/payoutsEnabled/status==='active' AND tiers.length>0, plus enableSubscriptions) and consume it everywhere monetisation UI renders: (1) monetisation page — disable Create-tier button + show dormant-until-Connect note; (2) org landing (space) — fold hasActiveTiers && subscriptionsEnabled into shouldShowSubscribeCTA so the SubscribeCTA banner and SubscribeStickyBar both suppress; (3) subscribers empty-state — branch on !connectReady → 'set up payments' vs !hasTiers → 'create a tier' vs true-empty. The data (getConnectStatus, listTiers/subscriptionContext, enableSubscriptions) is already reachable on all three surfaces; no new server round-trip needed on the write/studio surfaces.
- **First-creator / prerequisite-unaware empty states across the studio monetisation surfaces. Subscribers, Sales, Payouts, and the CreatorBreakdownRail all render generic 'nothing here yet' copy that assumes the only missing thing is customers, never surfacing the actual prerequisite chain (Connect → tiers → sales/payouts) or the solo-owner fee/creator-share context.** → `subscribers-empty-state`, `sales-payouts-empty-state-and-layout`
  - Shared fix: Apply one consistent condition-aware EmptyState convention (reuse the payouts-page <a class="empty-link"><Button/></a> pattern and the EmptyState action snippet) across subscribers, sales, payouts, and CreatorBreakdownRail: distinguish 'Connect not set up' vs 'no tiers' vs 'no activity yet', cross-link to /studio/monetisation (and revenue-share/team where relevant), and explain the fee/creator-share split + solo-owner-is-sole-beneficiary case. No new component needed.
- **Prod secret/config mismatch vs the documented TEST-mode launch bar. Stripe mode is derived entirely from the key prefix, so the sk_live_ key in ecom-api-production and the per-mode webhook signing secrets are a single config surface. The forgot-password endpoint drift is a parallel 'code assumes a literal never validated against the deployed dependency' class of defect but is fixed in code, not config.** → `connect-live-vs-test`
  - Shared fix: Rotate the production STRIPE_SECRET_KEY to sk_test_* AND swap the six production STRIPE_WEBHOOK_SECRET_* values to the corresponding TEST-mode endpoints' whsec_ (they are per-mode; a live secret rejects test-mode account.updated events and silently breaks onboarding-completion sync). Re-run deploy-production.yml and update the SECRETS_AND_VARS.md doc so the config matches intent. Optionally file a separate boot-time key/mode assertion bead.

### 2. `content-500-liberty` — 500 on audio content page /content/liberty
**Severity:** P0 · **Effort:** S · **Confidence:** high · **Area:** content-media

**Root cause:** The content-detail server load renders the TipTap body via `renderContentBody()`, which calls `generateHTML(content.contentBodyJson, ...)` imported from the WRONG entry point: `@tiptap/html` (the BROWSER build) instead of `@tiptap/html/server` (the Node build). The web app runs SSR on the Cloudflare Workers runtime (adapter-cloudflare / workerd). The compiled browser `generateHTML` starts with `if (typeof window === "undefined") throw new Error("generateHTML can only be used in a browser environment...use the @tiptap/html/server import instead.")`. `window` is not polyfilled by `nodejs_compat`, so this throws on every SSR call that reaches the JSON branch. The throw is unhandled: `renderContentBody()` has no try/catch, and `+page.server.ts:71` awaits it with no `.catch()` (and BEFORE any `setHeaders`), so the rejection propagates out of the load → the org route error boundary renders the generic 500 (matching the fetched page: full org shell "of blood and bones / powered by Revelations" wrapping a 500). Why ONLY /content/liberty on this org 500s: `liberty` is the unique item that is both `accessType:'free'` (so `isPublicAccessType('free')===true` triggers the eager `publicBodyHtml = await renderContentBody(content)` at line 71 for anonymous visitors) AND has `contentBodyJson` present (so `renderContentBody` takes the `generateHTML` branch rather than returning null). Verified against prod: liberty = audio/free/priceCents 0/contentBodyJson present; the other items are either paid+written with no contentBodyJson (body render skipped or returns null) or free video with no body JSON. The bug is not truly audio-specific — it is "public/free content that has a TipTap JSON body," which on this org is only the audio item. Note the markdown fallback branch (`marked`) is unaffected because it does not touch the DOM. Note also `@tiptap/html/server`'s guard is `isNode = process.versions.node != null` plus happy-dom; both are satisfied under the web worker's `nodejs_compat` flag (compatibility_flags: ["nodejs_compat"] in wrangler.jsonc), so the /server import is the correct runtime-compatible entry.

**Evidence:**
- `https://of-blood-and-bones.revelations.studio/content/liberty` — WebFetch returns HTTP 500 'Something went wrong' rendered inside the org shell (branding resolved), confirming the org layout load succeeded and the failure is in the content-detail page load/render, surfaced via the _org error boundary.
- `https://content-api.revelations.studio/api/content/public/discover` — liberty = contentType:audio, accessType:free, priceCents:0, contentBodyJson PRESENT. Other org items are paid+written with contentBodyJson ABSENT, or free video with no body JSON — so liberty is the only free item with a TipTap body, i.e. the only one that reaches generateHTML on the public/anonymous path.
- `apps/web/src/lib/editor/render.ts:20` — const { generateHTML } = await import('@tiptap/html'); — imports the BROWSER entry, not @tiptap/html/server. No try/catch around generateHTML.
- `node_modules/.pnpm/@tiptap+html@3.22.0/.../dist/index.js (generateHTML)` — Browser build: `function generateHTML(doc, extensions){ if (typeof window === 'undefined') throw new Error('generateHTML can only be used in a browser environment...use the @tiptap/html/server import instead.'); ... }` — throws on workerd where window is undefined.
- `node_modules/.pnpm/@tiptap+html@3.22.0/.../dist/server/index.js (generateHTML)` — Server build guards on `isNode = typeof process !== 'undefined' && process.versions.node != null` and serializes via happy-dom Window — the runtime-correct entry under nodejs_compat.
- `apps/web/src/routes/_org/[slug]/(space)/content/[contentSlug]/+page.server.ts:70-71` — const isPublic = isPublicAccessType(content.accessType); const publicBodyHtml = isPublic ? await renderContentBody(content) : null; — awaited with no .catch(), before any setHeaders; a throw here propagates to a 500.
- `apps/web/src/lib/server/content-detail.ts:151-155` — isPublicAccessType returns true only for accessType==='free', explaining why free liberty eagerly renders the body for anonymous visitors while paid written items do not.
- `apps/web/wrangler.jsonc` — compatibility_flags: ['nodejs_compat'] — process.versions.node is polyfilled (so @tiptap/html/server works) but window is NOT (so the current @tiptap/html browser build throws).
- `apps/web/svelte.config.js:2` — import adapter from '@sveltejs/adapter-cloudflare' — SSR runs on workerd, confirming the browser-vs-server entry mismatch is the failure mode.

**Proposed fix:** Primary fix (one line): in apps/web/src/lib/editor/render.ts change `const { generateHTML } = await import('@tiptap/html');` to `const { generateHTML } = await import('@tiptap/html/server');`. This entry uses the process.versions.node guard + happy-dom (both available under nodejs_compat), so TipTap JSON bodies render correctly during Cloudflare Workers SSR instead of throwing. `getRenderExtensions('full')` returns only StarterKit + Link (no browser-only extensions), which happy-dom serializes fine. Defense-in-depth (recommended, small): also make the render non-fatal so a body-render failure can never take down the whole page — either wrap the generateHTML branch in render.ts in try/catch returning null, or add `.catch(() => null)` to the two `await renderContentBody(content)` call sites (line 71 for public content, and line 153 for the gated authenticated path) in +page.server.ts, and mirror the same in the _creators variant (apps/web/src/routes/_creators/[username]/content/[contentSlug]/+page.server.ts) which uses the identical renderContentBody helper. The primary import fix is the actual bug; the catch is cheap insurance consistent with the loader's existing 'catch everything, degrade gracefully' pattern.

**Files to change:** `apps/web/src/lib/editor/render.ts`, `apps/web/src/routes/_org/[slug]/(space)/content/[contentSlug]/+page.server.ts`, `apps/web/src/routes/_creators/[username]/content/[contentSlug]/+page.server.ts`

**Open questions:**
- Confirm happy-dom is acceptable in the production SSR bundle size/CPU budget for the web worker (it is already a transitive dep of @tiptap/html, so no new top-level dependency is required). If happy-dom is considered too heavy for workerd, an alternative is a lightweight ProseMirror-JSON→HTML serializer, but @tiptap/html/server is the officially-sanctioned server path and works under nodejs_compat.
- Should the loader treat a body-render failure as a soft-degrade (render page without body) or should such content be considered broken? Recommend soft-degrade (return null body) so a malformed TipTap doc never 500s the whole content page.


### 3. `subscribe-component-shows-without-subscription` — Subscribe callout shows on org home with no subscription/Stripe
**Severity:** P1 · **Effort:** S · **Confidence:** high · **Area:** studio-ui

**Root cause:** On the org (space) landing page, both the inline `SubscribeCTA` banner and the bottom-of-viewport `SubscribeStickyBar` are gated solely by `shouldShowSubscribeCTA`, which is defined as `!!data.org?.id && accessChecked && !hasOrgAccess` (apps/web/src/routes/_org/[slug]/(space)/+page.svelte:230-232). This is only an "does the visitor already have access?" check — it flips on for every logged-out visitor (accessChecked becomes true immediately for anonymous users, +page.svelte:240-245) and for any logged-in non-member/non-subscriber. It NEVER checks whether the org actually has anything to subscribe to. There is no test for (a) active tiers, (b) the `enableSubscriptions` feature flag, or (c) Stripe Connect readiness.
