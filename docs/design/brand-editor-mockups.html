The three failures a redesign must fix

1Occlusion & scope. The panel covers the preview, and the preview is only ever the landing page — never grids, detail pages, the player, or forms.
2No map. Back-button-only across 12 levels — no breadcrumb, search, or jump. Getting from Colours to Typography means returning Home first.
3One width, every task. Picking a brand colour and tuning a shader's caustics share the same 360 px column.

Four directions

A Refined Panel — evolve today's panel; lowest risk.
B Studio Canvas — rail + full live canvas. Near-term recommendation.
C Inspector — tree → canvas → contextual inspector. The per-component future.
D Guided — presets & seed-first for non-designers.

How to read this

Each tab is a working mock, not a wireframe.

Switch tabs (or press 1–5). The Brand colour and theme controls in the top bar drive every preview at once, so you can judge each layout against the same live surface. Concepts B and C share one spine — a navigable structure beside a live canvas — so B can be built such that C is an additive step, not a rewrite.

Direction A · lowest-risk evolution

Refined Panel — the same box, unlost.

If re-architecting isn't on the table yet, the floating panel can still lose its worst habits. Dock it to the edge so it stops covering the page; give it a real breadcrumb, a search box, and inline accordion sections instead of a back-button maze; and add per-field reset plus a change review. Ships inside the current architecture — same store, same injection — with no backend work.

2 changes · Review & reset
1Docked, not floating. A dock-left / dock-right / float control parks the panel at the edge so it stops covering the surface it previews. (Float stays available for spot checks.)
2Breadcrumb + search. The existing getBreadcrumb() helper is finally rendered as a real trail, and a search box jumps straight to any control — no returning Home to cross between siblings.

4Review & reset. A change ledger with per-field revert replaces the all-or-nothing Reset. Same store, same injection path — no backend change.

Direction B · recommended near-term

Studio Canvas — the preview is the workspace.

Promote the editor out of a floating box into a dedicated Studio workspace: a fixed control rail beside a large live canvas. The canvas renders the org's real surfaces and you flip it between routes, devices, and themes — so you finally see your brand where it actually lives, at full size, nothing occluded. Try the route and device switchers; the seed control still re-themes it live.

Direction C · the extensibility target

Inspector — foundations today, components tomorrow.

The same rail-beside-canvas spine as B, with a third pane: a navigable tree on the left (Foundations → Surfaces → Components), the live canvas in the middle, and a contextual inspector on the right. Select a token group or a specific component — pick "Content card" in the tree, or click the highlighted card in the canvas — and the inspector shows exactly the properties that element honours. Per-component theming isn't a rebuild: the token layer already carries component-scoped keys (card-*, player-*, glass-tint, hero-*). This is the UI that would surface them.

1Two ways in. Navigate the tree, or click an element in the canvas — both open the same inspector. Direct manipulation is how designers expect to work.
2Grounded in real tokens. Every inspector row shows the CSS variable it writes — --brand-card-hover-scale, --media-scrim — keys the engine already honours. Adding a component is a "Path B" change: no migration.
3Overrides layer, never fork. A component tweak sits on top of the derived palette and always carries "Reset to derived", so per-component edits can't strand an org off-brand.

What it would take to ship per-component editing (future, not now)

1. A component registry — map each component (Button, Content card, Nav, Player, Hero, Form) to the override keys it honours, with safe ranges and sensible controls.

2. Tree/inspector IA — the structure shown here, so "Components" is a first-class branch beside "Foundations".

3. Canvas selection — a lightweight data-brand-component annotation on component roots so a click maps to the registry.

4. Guardrails — overrides layer on the derived palette (never replace it), each with a visible reset. Contrast checks travel with the component.

Direction D · the 60-second path

Guided — good-looking before it's deep.

Most admins want their space to look on-brand fast, not to tune a shader. This entry mode leads with three fast starts — pick a preset, drop a seed colour and get complete palettes, or pull colours from your logo — with a live preview the whole time. "Go deeper" hands off to the full editor (B) only when they want it. It directly answers the v2 note: one click to a complete palette, not a field-by-field grind.
